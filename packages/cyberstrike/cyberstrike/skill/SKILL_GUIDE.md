# SKILL.md Authoring Guide

This guide defines the structure, conventions, and quality standards for creating CyberStrike skill files.

## Directory Structure

```
.cyberstrike/skill/
├── SKILL_GUIDE.md                          # This file
├── wstg-inpv-05/SKILL.md                  # OWASP WSTG skills (flat)
├── wstg-athn-03/SKILL.md
├── CIS_benchmarks/                         # Compliance frameworks
│   └── AWS/
│       ├── foundations-v7/                 # CIS AWS Foundations Benchmark v7.0.0
│       │   ├── 2.1.1/SKILL.md
│       │   └── ...
│       ├── compute/
│       ├── database/
│       ├── security/
│       └── storage/
├── ad-security/SKILL.md                    # Custom/community skills
└── kerberos-attacks/SKILL.md
```

The skill loader recursively scans `**/**/SKILL.md` — any depth is supported.

---

## Frontmatter Schema

Every SKILL.md starts with YAML frontmatter between `---` markers.

### Required Fields

| Field         | Type   | Description                                        |
| ------------- | ------ | -------------------------------------------------- |
| `name`        | string | Unique skill identifier, must match directory name |
| `description` | string | One-line summary of what this skill tests/checks   |

### Optional Fields

| Field            | Type     | Description                                                   |
| ---------------- | -------- | ------------------------------------------------------------- |
| `category`       | string   | Domain category (see Categories below)                        |
| `version`        | string   | Semver version (e.g., `"1.0.0"`)                              |
| `author`         | string   | Author identifier (e.g., `cyberstrike-official`)              |
| `tags`           | string[] | Searchable keywords                                           |
| `owasp_id`       | string   | OWASP test ID (e.g., `WSTG-INPV-05`)                          |
| `cis_id`         | string   | CIS benchmark ID (e.g., `2.1.1`)                              |
| `cis_benchmark`  | string   | CIS benchmark name + version                                  |
| `tech_stack`     | string[] | Technologies this skill applies to                            |
| `cwe_ids`        | string[] | Related CWE identifiers                                       |
| `chains_with`    | string[] | Skills that chain with this one (kill chain)                  |
| `prerequisites`  | string[] | Skills that should run before this one                        |
| `severity_boost` | map      | `skill_name: "explanation"` — why chaining increases severity |

### Signing Fields (auto-generated by `cyberstrike skill sign`)

| Field       | Type   | Description       |
| ----------- | ------ | ----------------- |
| `sha256`    | string | Content hash      |
| `signature` | string | Ed25519 signature |
| `signed_by` | string | Signer identifier |

## Categories

### Offensive / Pentest (OWASP WSTG)

| Category                | Description                                |
| ----------------------- | ------------------------------------------ |
| `information-gathering` | Recon, fingerprinting, enumeration         |
| `input-validation`      | SQLi, XSS, command injection, etc.         |
| `authentication`        | Login, session, credential testing         |
| `authorization`         | IDOR, privilege escalation, access control |
| `client-side`           | DOM XSS, clickjacking, WebSocket           |
| `business-logic`        | Logic flaws, workflow bypass               |
| `configuration`         | HTTP methods, headers, TLS, error handling |
| `cryptography`          | Weak ciphers, key management               |
| `error-handling`        | Stack traces, verbose errors               |
| `api-testing`           | REST, GraphQL, WebSocket API testing       |
| `identity`              | SSO, OAuth, SAML, federation               |

### Compliance (CIS Benchmarks)

| Category         | Description                            |
| ---------------- | -------------------------------------- |
| `cis-iam`        | Identity and Access Management         |
| `cis-storage`    | S3, EBS, EFS storage security          |
| `cis-logging`    | CloudTrail, Config, flow logs          |
| `cis-monitoring` | Alarms, Security Hub, event monitoring |
| `cis-networking` | VPC, security groups, NACLs, endpoints |

## Content Structure

### For Offensive/Pentest Skills (OWASP)

```markdown
# Skill Title

## High-Level Description

Brief explanation of the vulnerability class and why it matters.

## What to Check

- [ ] Checklist of things to verify
- [ ] Organized by sub-area

## How to Test

### Step 1: Title

Code blocks with actual commands/scripts the agent can reference.

### Step 2: Title

More detailed testing procedures.

## Tools

| Tool | Purpose | Usage |
| ---- | ------- | ----- |

## Example Commands/Payloads

Ready-to-use payloads and scripts.

## Remediation Guide

Fix examples in relevant languages/frameworks.

## Risk Assessment

| Finding | CVSS | Severity |
| ------- | ---- | -------- |

## CWE Categories

| CWE ID | Title |
| ------ | ----- |

## References

- Links to authoritative sources

## Checklist

Final verification checklist.
```

### For Compliance Skills (CIS Benchmarks)

```markdown
# CIS Control Title

## Description

What this control checks and why it matters.

## Rationale

Business/security justification for this control.

## Impact

What changes when you enable/enforce this control.

## Audit Procedure

### Using AWS CLI

Exact commands to check compliance status.

### Using AWS Console

Step-by-step console navigation (for manual checks).

## Expected Result

What a compliant configuration looks like.

## Remediation

### Using AWS CLI

Commands to fix non-compliant configurations.

### Using AWS Console

Console steps to remediate.

## Default Value

What AWS sets by default (compliant or not).

## References

- CIS Controls mapping
- AWS documentation links

## Profile

Level 1 or Level 2, and assessment status (Automated/Manual).
```

## Quality Standards

### DO

- **Write for an AI agent** — the agent reads this at runtime to understand what to do
- **Include real commands** — `aws cli`, `curl`, `nmap` commands the agent can reference or adapt
- **Include error patterns** — regex patterns, response codes, strings that indicate a finding
- **Include severity context** — CVSS scores, CWE mappings, risk ratings
- **Include remediation** — the agent may need to explain fixes in a pentest report
- **Keep payloads ready-to-use** — the agent can adapt them to the target
- **Use tables for structured data** — easier for the agent to parse
- **Reference tools the agent has** — MCP servers (hackbrowser, cloud-audit, etc.)

### DON'T

- **Don't add decision flow logic (STEP/WHEN/GOTO)** — the AI model does its own reasoning
- **Don't strip useful content for "token savings"** — lazy loading means only 1 skill loads at a time
- **Don't duplicate content across skills** — if two skills share context, use `chains_with` to link them
- **Don't add generic security theory** — the model already knows this
- **Don't hardcode target URLs** — use `target.com` or `{target}` as placeholders

## Naming Conventions

| Skill Type          | Name Pattern                    | Example                     |
| ------------------- | ------------------------------- | --------------------------- |
| OWASP WSTG          | `wstg-{category}-{number}`      | `wstg-inpv-05`              |
| CIS AWS Foundations | `cis-aws-foundations-{section}` | `cis-aws-foundations-2.1.1` |
| CIS AWS Compute     | `cis-aws-compute-{section}`     | `cis-aws-compute-3.1`       |
| Custom              | `{descriptive-name}`            | `kerberos-attacks`          |

The `name` field in frontmatter must be unique across all skills.

## Frontmatter Examples

### OWASP Pentest Skill

```yaml
---
name: wstg-inpv-05
description: "Testing for SQL Injection"
category: input-validation
owasp_id: WSTG-INPV-05
version: "1.0.0"
author: cyberstrike-official
tags: [injection, sqli, database]
tech_stack: [mysql, postgresql, mssql, oracle, sqlite]
cwe_ids: [CWE-89, CWE-564]
chains_with: [wstg-authz-02, wstg-inpv-06]
prerequisites: [wstg-info-01]
severity_boost:
  wstg-authz-02: "SQLi + IDOR = Account Takeover (Critical)"
---
```

### CIS Benchmark Skill

```yaml
---
name: cis-aws-foundations-2.1.1
description: "Ensure centralized root access in AWS Organizations"

author: cyberstrike-official
tags: [cis, aws, iam, organizations, root-access]
cis_id: "2.1.1"
cis_benchmark: "CIS AWS Foundations Benchmark v7.0.0"
tech_stack: [aws]
cwe_ids: []
chains_with: []
prerequisites: []
severity_boost: {}
---
```

## Testing Your Skill

```bash
# Verify skill is discovered
cd packages/cyberstrike && bun run dev
# Then in the TUI:
#   /skill list
#   /skill search <keyword>

# Verify frontmatter parses correctly
# The skill loader will log errors for malformed skills

# Sign the skill (official skills only)
cyberstrike skill sign <skill-name>
```
